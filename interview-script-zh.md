# CareLoop 面试陈述脚本：AI 如何赋能产品 MVP 设计落地

## 1 分钟开场

CareLoop 是一个面向长期服药慢病用户的治疗路径守护原型。

我没有一开始做完整医疗 App，而是先拆出最高频、最关键的用户场景：
用户在提醒出现的一刻，需要用最低成本记录“已服用、稍后提醒、还是不确定”。

所以这个 MVP 的核心闭环是：

锁屏 / widget 提醒 -> 一键记录状态 -> 安全处理“不确定是否服药” ->
识别治疗路径脱落风险 -> AI 生成复诊前问题清单。

这里的 AI 不是医生，不做诊断、不改药、不提供补服建议。AI 的作用是把用户自记录的碎片数据整理成医生沟通材料，降低复诊准备成本。

## AI 如何帮助我设计 MVP

我把 AI 当作产品共创工具，而不是只当代码生成器。

我的流程是：

1. 用 AI 一起澄清用户问题，避免一上来做过大的完整 MVP。
2. 把产品拆成一个可展示、可验证的核心闭环。
3. 明确哪些部分今天必须做，哪些部分先 mock。
4. 用 AI coding 快速生成和迭代可点击原型。
5. 再把原型升级成能讲清楚产品策略、AI 边界、指标和 roadmap 的作品。

AI 在这个过程中最大的价值是帮我快速压缩不确定性：从模糊想法到页面结构、交互路径、文案边界、再到可运行 demo。

## 为什么重点做 widget 和锁屏

对于长期服药用户，问题不只是有没有一个漂亮 dashboard。

真正的摩擦点是：提醒出现的那一刻，用户愿不愿意顺手记录。

所以我把入口放在 widget 和锁屏弹窗，而不是让用户每天主动打开 App。

这个判断对应的是产品取舍：先降低记录成本，再谈更复杂的健康管理功能。

## AI 如何嵌入产品

我把 AI 嵌入设计成三层：

第一层是输入：
用户自记录的数据，比如漏服、不确定、未确认、疲劳评分变化、距离复诊还有多少天。

第二层是边界：
模型必须明确不做诊断、不建议改药、不提供补服医疗建议。

第三层是输出：
AI 只生成复诊沟通问题，比如：

- 我这个月有两次不确定是否服药，未来应该怎么处理？
- 我漏了一次每周药，下次是否需要联系诊所？
- 最近疲劳评分升高，是否需要讨论补充检查？
- 下次复查时间或检查安排是否需要调整？

当前面试 demo 默认使用 mock AI 输出。这样做是为了保证展示稳定，也避免医疗场景里模型输出不确定带来的风险。

真实上线时，我会通过后端或 Vercel serverless function 调用模型，而不是把 API key 放在前端。

## DeepSeek 和 Gemini 怎么选

如果面试 DeepSeek，我会优先用 DeepSeek 来讲产品架构，因为它和岗位更相关，而且适合展示我对对话模型、API 兼容和 Agent 工具链的理解。

如果只是为了快速验证，我也可以用 Gemini Pro，因为我已经有可用额度。

但我的架构不会绑定死某一个模型。前端只调用自己的 `/api/generate-questions`，后端通过环境变量切换 DeepSeek 或 Gemini。

这样产品上更稳：模型供应商可以替换，安全规则可以集中管理，API key 也不会泄露到前端。

## 我会看的指标

我不会只看页面是否好看，还会看行为数据：

- 提醒确认率
- Not sure 事件比例
- 每周药完成率
- AI 复诊问题生成率
- D7 留存
- 开启二次提醒后的确认率变化
- AI 输出被接受、编辑、忽略的比例

这些指标能判断两个核心假设：

1. widget / 锁屏入口是否真的降低记录成本；
2. AI 复诊准备是否真的让用户觉得有用。

## Roadmap

当前 prototype：
验证核心闭环和产品边界。现在版本还加入了 localStorage 和事件日志，
点击已服用、不确定、稍后提醒后会沉淀为本地事件，并驱动 Log 和 PM
页面里的指标展示。

V1：
做原生通知、锁屏 widget、安全本地存储、事件埋点和数据看板。

V2：
接入 DeepSeek 或 Gemini，做结构化复诊摘要、医生沟通模板、家庭成员协作和导出功能。

暂不做：
诊断、改药、补服建议、自动医疗决策。

## 结尾

这个项目想证明的不是“我一天做完了完整医疗 App”。

我想展示的是：我能用 AI 工具把一个模糊需求快速拆成可验证的产品闭环，并且在设计里同时考虑用户体验、模型边界、数据指标、工程落地和后续迭代。
